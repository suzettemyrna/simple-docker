## Part 2. Операции с контейнером

> English version: [Part2.md](../eng/Part2.md)

### 2.1. Просмотр конфигурации Nginx

Чтобы посмотреть содержимое конфигурационного файла внутри работающего контейнера, выполним команду:

```bash
docker exec -it <container_id|container_name> cat /etc/nginx/nginx.conf
```

где:

* `docker exec` — выполнение команды внутри работающего контейнера;
* `cat` — вывод содержимого файла.

Выводится содержимое файла `nginx.conf` из контейнера.

![docker exec](../../images/part2/1.png)

---

### 2.2. Создание собственного файла nginx.conf

Создадим файл `nginx.conf` на локальной машине и настроим в нём страницу статуса сервера по адресу `/status`.

![nginx.conf](../../images/part2/2.png)

Основные директивы:

* `http {}` — блок настроек HTTP;
* `server {}` — описание веб-сервера;
* `listen 80;` — прослушивание входящих подключений на порту 80;
* `location /status {}` — обработка запросов по адресу `/status`;
* `stub_status on;` — включение страницы со статистикой Nginx.

---

### 2.3. Копирование конфигурации в контейнер

Скопируем созданный файл внутрь контейнера, заменив им старый файл конфигурации:

```bash
docker cp nginx.conf <container_id|container_name>:/etc/nginx/nginx.conf
```

![docker cp](../../images/part2/3.png)

---

### 2.4. Применение новой конфигурации

Чтобы применить новую конфигурацию, перезагрузим Nginx внутри контейнера:

```bash
docker exec <container_id|container_name> nginx -s reload
```

![docker exec](../../images/part2/4.png)

---

### 2.5. Проверка страницы статуса

Откроем в браузере адрес:

```text
http://localhost/status
```

![localhost/status](../../images/part2/5.png)

Страница отображает статистику работы Nginx.

Основные параметры:

* `Active connections` — количество активных соединений;
* `accepts` — количество принятых соединений;
* `handled` — количество обработанных соединений;
* `requests` — количество обработанных HTTP-запросов;
* `Reading` — соединения, из которых читаются данные;
* `Writing` — соединения, которым отправляется ответ;
* `Waiting` — соединения в режиме ожидания.

---

### 2.6. Экспорт контейнера

Экспортируем файловую систему контейнера в архив с названием `container.tar`:

```bash
docker export <container_id|container_name> > container.tar
```

![container.tar](../../images/part2/6.png)

---

### 2.7. Остановка контейнера

Остановим контейнер:

```bash
docker stop <container_id|container_name>
```

![docker stop](../../images/part2/7.png)

---

### 2.8. Удаление образа

Удалим образ из локального хранилища Docker, не удаляя при этом контейнер:

```bash
docker rmi <image_id|repository>
```

![docker rmi](../../images/part2/8.png)

---

### 2.9. Удаление контейнера

Удалим остановленный контейнер:

```bash
docker rm <container_id|container_name>
```

Проверим успешное удаление контейнера с помощью команды:

```bash
docker ps -a
```

Данная команда отображает все существующие контейнеры, включая остановленные.

![docker rm](../../images/part2/9.png)

---

### 2.10. Импорт контейнера

Создадим новый образ из ранее экспортированного архива:

```bash
docker import \
-c 'cmd ["nginx","-g","daemon off;"]' \
-c 'ENTRYPOINT ["/docker-entrypoint.sh"]' \
container.tar nginx
```

где:

* `CMD` задаёт команду, которая будет выполнена при запуске контейнера;
* `ENTRYPOINT` определяет точку входа контейнера;
* `container.tar` — импортируемый архив;
* `nginx` — имя нового образа.

Команда возвращает хеш (SHA256) нового образа.

![docker import](../../images/part2/10.png)

---

### 2.11. Запуск импортированного образа

Создадим и запустим контейнер на основе импортированного образа:

```bash
docker run -d -p 80:80 nginx
```

Проверим статус контейнера:

```bash
docker ps
```

![docker run](../../images/part2/11.png)

---

### 2.12. Повторная проверка страницы статуса

Откроем страницу:

```text
http://localhost/status
```

![localhost/status](../../images/part2/12.png)

Страница статуса по-прежнему доступна после экспорта, удаления и повторного создания образа и контейнера.

---

## Итог

Изменена конфигурация Nginx внутри контейнера, выполнены экспорт и импорт контейнера через архив. После восстановления образа и контейнера работоспособность сервиса сохранена.
